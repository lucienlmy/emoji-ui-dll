# 二级菜单修复技术报告

## 1. 背景

本次修复针对 `examples/Python/demo_all_components_tabs.py` 中两类多级菜单问题：

1. 窗口右键菜单的二级菜单可以显示，但点击无回调。
2. 顶部 `文件` 菜单的一级弹层存在，二级菜单在部分场景下无法点击，后续修复过程中又出现了悬停 `跳转页签` 时进程闪退的问题。

本次修复目标不是仅让菜单“显示出来”，而是同时解决以下几点：

1. 二级菜单能够稳定展开。
2. 二级菜单项点击后能够触发 C++ -> Python 回调。
3. 父菜单、子菜单之间移动时不提前消失。
4. 不因菜单销毁链导致崩溃。

## 2. 问题现象

修复前实际表现如下：

1. 一级菜单悬停到带子项的条目后，视觉上出现二级菜单，但点击 `取消/确定` 或普通子项时没有任何回调。
2. 用户点击二级菜单区域时，日志显示消息仍然落在主窗口，未进入子菜单窗口。
3. 部分场景下，鼠标准备移入二级菜单时，父菜单会提前关闭。
4. 在把二级菜单改为复用 `PopupMenu` 链后，顶部 `文件 -> 跳转页签` 悬停会触发崩溃，整个窗口被销毁。

## 3. 根因分析

### 3.1 点击没有进入二级菜单窗口

原始实现中，一级弹层和二级弹层走的是两套不同机制：

1. 一级菜单使用 `PopupMenuState + PopupMenuProc`。
2. 二级菜单使用单独的 `SubMenuWindow` 类。

主窗口为了兼容“点击被主窗口吃掉”的情况，增加了 `DispatchVisiblePopupMenuClick(...)` 手动转发逻辑。但这套逻辑最初只稳定处理一级 `PopupMenu`，对 `SubMenuWindow` 的命中和生命周期把控不够稳定，导致：

1. 二级菜单虽然显示。
2. 但点击没有稳定路由到 `SubMenuWindow::OnClick(...)`。
3. 最终表现为“看得到，点不到，没有回调”。

### 3.2 父子菜单之间存在跨窗口移动区

一级菜单和二级菜单不是同一个窗口，鼠标从父菜单项移动到子菜单时，会经过一个短暂的“过渡区域”。如果父菜单只根据自身客户区命中判断悬停状态，就会误判为鼠标已离开，提前关闭子菜单。

因此需要：

1. 识别父项矩形与子菜单矩形之间的过渡区域。
2. 在鼠标经过该区域时，保持父项悬停状态。

### 3.3 旧的子菜单销毁链复杂

早期修复里，二级菜单既可能由 `SubMenuWindow` 持有，也可能被手动转发命中。随着补丁叠加，生命周期控制变复杂，维护成本升高。

为了提高稳定性，后续将二级菜单实现收敛到与一级菜单相同的 `PopupMenu` 体系。

### 3.4 闪退根因是二次释放

把二级菜单改成 `PopupMenu` 链后，新增了 `child_popup_handle` / `parent_popup_handle` 关联关系。此时关闭子菜单会进入：

1. `DestroyEmojiPopupMenu(...)`
2. `DestroyWindow(...)`
3. `WM_DESTROY` 清理 `PopupMenuState`

但旧的 `DestroyEmojiPopupMenu(...)` 在窗口存在时，`DestroyWindow(...)` 之后仍继续手动：

1. 从 `g_popup_menus` 擦除
2. `delete state`

而 `WM_DESTROY` 中也会做清理和 `delete state`，于是形成双重释放。这正是 `文件 -> 跳转页签` 悬停时闪退的直接原因。

## 4. 修复过程

### 4.1 第一阶段：补齐一级菜单点击转发链

在主窗口中加入对可见弹出菜单的点击分发：

1. 新增 `DispatchVisiblePopupMenuClick(...)`
2. 新增 `ActivatePopupMenuItem(...)`
3. 对 `WM_LBUTTONDOWN / WM_LBUTTONUP` 做手动转发

目的：

1. 当弹层未直接收到点击时，由主窗口按屏幕坐标命中可见 popup。
2. 保证一级菜单至少可以稳定展开和触发父级回调。

### 4.2 第二阶段：修复焦点丢失导致的提前关闭

为了避免点击子菜单前父菜单被焦点变化提前销毁，关闭了若干依赖焦点切换自动隐藏的路径：

1. `PopupMenuProc` 中不再在 `WM_KILLFOCUS / WM_ACTIVATE(WA_INACTIVE)` 时立即关闭。
2. `SubMenuWindow` 中也不再依赖失焦立刻 `Hide()`。

目的：

1. 让菜单关闭更多地由显式点击逻辑控制。
2. 避免主窗口抢消息时先把菜单销毁。

### 4.3 第三阶段：修正父菜单到子菜单的坐标换算

一级菜单的菜单项边界 `item.bounds` 是按 D2D 逻辑单位保存的，而子菜单窗口位置最终需要屏幕像素坐标。

因此修复了：

1. `ShowPopupSubMenu(...)` 中 `item.bounds` 到屏幕坐标的 DPI 换算。
2. `IsPointInsidePopupSubMenuTransit(...)` 中父项矩形到屏幕坐标的 DPI 换算。

目的：

1. 让二级菜单实际窗口矩形与视觉位置一致。
2. 避免“看上去在这里，实际命中区域偏掉”的问题。

### 4.4 第四阶段：将二级菜单切换为复用 PopupMenu 体系

这是本次修复中最关键的结构性调整。

最终没有继续深挖 `SubMenuWindow` 作为一级菜单二级弹层的实现，而是改为：

1. 在 `PopupMenuState` 中新增 `child_popup_handle`
2. 在 `PopupMenuState` 中新增 `parent_popup_handle`
3. `ShowPopupSubMenu(...)` 不再创建 `SubMenuWindow`
4. 而是直接创建新的 `PopupMenuState` 作为子 popup

这样处理后：

1. 一级菜单与二级菜单统一走 `PopupMenuProc`
2. 绘制机制统一
3. 点击命中统一
4. 回调链统一
5. 父子级销毁关系也更清晰

### 4.5 第五阶段：修复 PopupMenu 双重释放崩溃

在 `DestroyEmojiPopupMenu(...)` 中加入关键保护：

1. 如果 `state->hwnd` 存在且窗口有效，只调用 `DestroyWindow(...)`
2. 随后立刻 `return`
3. 剩余资源释放全部交给 `WM_DESTROY`

这样可以确保：

1. `PopupMenuState` 只被销毁一次
2. `g_popup_menus` 只按统一路径清理
3. 顶部菜单悬停二级菜单时不再闪退

## 5. 最终方案说明

当前最终方案可以概括为：

1. 一级菜单和二级菜单统一使用 `PopupMenuState + PopupMenuProc`
2. 主窗口负责为“被主窗口抢到的点击”做兜底转发
3. 父菜单到子菜单之间保留过渡区域判断
4. 二级 popup 通过 `child_popup_handle` 和 `parent_popup_handle` 形成层级关系
5. popup 的资源释放统一走 `DestroyWindow -> WM_DESTROY`

这套方案相较旧的 `SubMenuWindow` 路径有两个直接优点：

1. 逻辑链更短，二级菜单不再是另一套独立机制。
2. 问题定位更简单，后续维护只需要围绕 `PopupMenu` 一套代码展开。

## 7. 关键技术点

### 7.1 手动点击转发

当 Windows 消息没有直接送达 popup 时，主窗口使用屏幕坐标再次命中可见 popup，并手动触发条目激活逻辑。

### 7.2 DPI 感知坐标换算

菜单项边界是 D2D 逻辑单位，窗口命中和定位是像素坐标，两者必须显式换算，否则二级菜单位置和命中区会错位。

### 7.3 父子菜单链

通过 `parent_popup_handle` 记录父级 popup，可以在叶子项点击后顺序关闭父级链，而不是只关闭当前子菜单。

### 7.4 单一路径释放

在带窗口句柄的 popup 场景下，只允许 `WM_DESTROY` 做最终清理，这是避免二次释放的关键。

## 8. 验证结果

本轮修复后，已确认：

1. 窗口右键菜单的二级菜单可以展开并触发回调。
2. 顶部 `文件` 菜单的二级菜单可以使用。
3. `文件 -> 跳转页签` 悬停不再因二次释放导致进程闪退。

## 9. 后续建议

虽然当前问题已修复，但从维护角度建议继续做两件事：

1. 逐步淡出一级菜单场景下对 `SubMenuWindow` 的依赖，避免两套二级菜单机制长期并存。
2. 为 `PopupMenuState` 增加更明确的父子销毁工具函数，减少未来继续叠补丁时再次引入生命周期错误的概率。

## 10. 结论

这次问题的本质不是“菜单没画出来”，而是多套菜单实现并存、消息路由不统一、再叠加释放路径不一致，最终导致：

1. 二级菜单点击不生效
2. 鼠标移动时菜单提前关闭
3. 修复过程中出现双重释放闪退

最终通过“统一二级菜单到 `PopupMenu` 体系、补齐主窗口点击转发、修正 DPI 坐标换算、收敛 popup 销毁路径”，问题已经被收敛到可维护的状态。
