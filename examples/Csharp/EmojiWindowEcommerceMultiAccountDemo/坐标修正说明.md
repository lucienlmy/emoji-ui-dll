# EmojiWindowEcommerceMultiAccountDemo 坐标修正说明

## 1. 目的

这份文档记录本项目里已经验证过的：

- 正确坐标规则
- 错误坐标写法
- 当前固定布局版 `EcommerceFixedLayoutApp.cs` 的基准坐标

重点不是“UI 审美说明”，而是防止后续继续把控件放飞、重叠、漂移。

---

## 2. 最关键结论

### 2.1 不同控件的坐标规则不一样

这次反复出错的根因，不是单纯数字写错，而是 **坐标参考系混用了**。

在当前 DLL / C# 封装里，大体遵循下面规律：

### A. 这些控件通常按“父容器相对坐标”放置

- `CreatePanel(...)`
- `CreateLabel(...)`
- `CreateEditBox(...)`
- `CreateComboBox(...)`
- `CreateDataGridView(...)`

也就是说：

- 父容器如果是 `_sidebarPanel`
- 子控件的 `(x, y)` 就应当相对于 `_sidebarPanel` 左上角

### B. 这些按钮类控件，最终要按“主窗口绝对坐标”去改位置

- `create_emoji_button_bytes(...)`
- `SetButtonBounds(...)`

这次验证下来的经验是：

- **按钮创建时虽然传了 parent**
- 但后续 `SetButtonBounds(...)` 调整位置时，更稳定的是按主窗口绝对坐标处理

如果把按钮当成 panel 内部相对坐标去调，很容易出现：

- 按钮飞到左上角
- 按钮压到别的控件上
- 按钮和输入框不在一个参考系里

## 3. 当前固定布局版基准

文件：

- `examples/Csharp/EmojiWindowEcommerceMultiAccountDemo/EcommerceFixedLayoutApp.cs`

当前基准窗口：

- 宽：`1440`
- 高：`900`

全局基准：

- `padding = 16`
- `gap = 12`
- `TitleBarHeight = 32`
- `topInset = 40`
- 顶部卡片区：`y = 40`
- 卡片高度：`96`
- 主内容起始：`contentY = 148`

## 4. 当前“正确坐标”基准

以下是当前固定布局版的有效基准。

### 4.1 顶部卡片

- 卡片区起点：`cardsY = 40`
- 卡片高度：`96`
- 每张卡片宽度：
  - `(_width - padding * 2 - gap * 5) / 6`

### 4.2 换肤按钮

当前正确位置：

- `x = _width - padding - 52`
- `y = cardsY + cardsHeight + 16`
- `w = 34`
- `h = 34`

说明：

- 这个按钮是 **主窗口绝对坐标**
- 不归任何 panel 的相对布局管理

### 4.3 左侧侧栏

侧栏 panel：

- `x = 16`
- `y = 148`
- `w = 332`

侧栏内部相对坐标：

- 三个筛选框：
  - `y = 44`
- 搜索框：
  - `y = 96`
- 第一排按钮：
  - `y = 146`
- 第二排按钮：
  - `y = 188`
- 表格标题：
  - `y = 236`
- 表格：
  - `y = 264`

### 4.4 左侧按钮

注意：按钮位置最终按 **主窗口绝对坐标** 调整。

当前按钮绝对坐标计算方式：

- 第一排按钮：
  - `x = padding + sx + ...`
  - `y = contentY + 146`
- 第二排按钮：
  - `x = padding + sx + ...`
  - `y = contentY + 188`

其中：

- `sx = 12`
- `contentY = 148`

所以：

- 第一排按钮基准 y 实际是 `294`
- 第二排按钮基准 y 实际是 `336`

### 4.5 右侧工作区

工作区 panel：

- `x = 360`
- `y = 148`
- `w = _width - 360 - 16`

右侧内部相对坐标：

- 标题：`y = 42`
- 地址栏：`y = 78`
- 主操作按钮：`y = 136`
- 浏览器容器：`y = 196`

### 4.6 右侧按钮

同样，按钮最终按 **主窗口绝对坐标** 调整更稳定。

例如：

- `进入`：
  - `x = rightX + wx + workW - urlBtnW`
  - `y = contentY + 78`

- 主操作按钮行：
  - `x = rightX + wx + ...`
  - `y = contentY + 136`

## 5. 错误写法总结

下面这些都是这次反复踩到的错误。

### 错误 1：把按钮当成 panel 相对坐标

错误示例：

```csharp
EmojiWindowNative.SetButtonBounds(_btnAdd, sx, 146, btnW, 34);
```

为什么错：

- 这会让按钮按主窗口左上角附近去摆
- 而不是按 `_sidebarPanel` 内部去摆

结果：

- 左侧按钮和筛选框重叠
- 按钮漂到 panel 外面

正确做法：

```csharp
EmojiWindowNative.SetButtonBounds(_btnAdd, padding + sx, contentY + 146, btnW, 34);
```

### 错误 2：同一层出现两套筛选/搜索

错误表现：

- 顶部一套筛选
- 左侧又一套筛选
- 顶部全局搜索
- 左侧又一套搜索

结果：

- 信息架构冲突
- 用户看不懂哪个才是真正用的

正确做法：

- 固定只保留一套侧栏筛选
- 顶部只保留卡片和换肤按钮

### 错误 3：隐藏控件但还保留旧布局逻辑

错误表现：

- 控件视觉上隐藏了
- 但仍然占着原来的布局推导逻辑
- 后面新控件又压过来

结果：

- 看起来像“坐标对了又不对”

正确做法：

- 视觉上不用的控件，直接 `-2000,-2000,1,1`
- 并且不要再让它参与当前层级布局判断

### 错误 4：在同一个函数里混用“相对坐标思维”和“绝对坐标思维”

这是这次最容易反复出现的问题。

正确处理方式：

- `ComboBox / EditBox / DataGrid / Label`
  - 先默认按父容器相对坐标
- `Button`
  - 统一按主窗口绝对坐标

不要在一个按钮组里一半写相对，一半写绝对。

### 错误 5：在旧布局函数上连续 patch

错误表现：

- 改一处好一点
- 下一次又被另一组坐标覆盖

结果：

- 越修越乱

正确做法：

- 要么单独做一个新布局函数
- 要么彻底替换旧布局逻辑
- 不要一直在旧逻辑上叠补丁

## 6. 后续改布局的建议顺序

如果后面还要继续调坐标，建议严格按这个顺序：

1. 先定窗口整体分区
   - 顶部卡片
   - 左侧侧栏
   - 右侧工作区
   - 底部状态栏

2. 再定 panel 的矩形

3. 再定 `ComboBox / EditBox / Label / DataGrid`

4. 最后单独定按钮

不要一边改 panel 一边改按钮。

---

## 7. 当前建议

如果以后还要继续微调：

- 只改 `EcommerceFixedLayoutApp.cs`
- 不要再回到 `EcommerceMultiAccountBrowserApp.cs`

原因：

- `EcommerceMultiAccountBrowserApp.cs` 已经堆了太多历史布局补丁
- `EcommerceFixedLayoutApp.cs` 才是当前可维护的固定布局版

---

## 8. 一句话总结

这次最重要的经验不是“某个按钮应该多往左 8 像素”，而是：

- **按钮用绝对坐标**
- **输入框/表格/标签优先按父容器相对坐标**
- **隐藏旧控件时别让它继续参与布局思维**
- **坏掉的布局函数不要继续 patch，应该另起一套固定布局**
